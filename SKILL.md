# Feishu Link Bot

## First Rule

- 先查官方文档和现成经验，再动手。
- 优先看两类资料：
  - OpenClaw 官方飞书通道文档
  - 飞书官方文档：自建应用、机器人能力、事件订阅、长连接、版本发布
- GitHub 和技术社区只用来补“实操坑位”和“常见误判”，不能覆盖官方结论。
- 不要一上来就做桥接、轮询、注入脚本或其他过渡方案，除非用户明确接受临时方案。

## Source Priority

- 先看 OpenClaw 官方文档，确认接入模式和本地 gateway 要求。
- 再看飞书官方文档，重点核对：
  - 企业自建应用开发流程
  - 创建并配置应用
  - 事件订阅与 `im.message.receive_v1`
  - 发布与审核自建应用
  - 配置文件导入/应用配置文件
- 最后再看 GitHub 和技术社区，主要用途是确认：
  - 大多数示例仍默认先在开放平台手动创建应用
  - `App ID/App Secret` 只是接入凭证，不等于应用侧已可用
  - UI 卡住时有哪些常见误判

## What App ID And Secret Can Solve

- 仅从本地 OpenClaw 配置角度看，`App ID` 和 `App Secret` 就够了。
- 但从“用户真的能在飞书里发消息并收到回复”的角度看，只给 `App ID` 和 `App Secret` 不够。
- 还必须同时满足：
  - 飞书应用是自建应用
  - 已启用机器人能力
  - 事件订阅方式是长连接
  - 已发布的线上版本包含 `im.message.receive_v1`
  - 相关权限已在目标租户生效
- 如果浏览器里已经有飞书开放平台的登录态，并且当前账号有创建与发布权限，则不一定要求“应用预先存在”。
- 但如果用户坚持“只给我 `App ID` 和 `App Secret`”，这句话本身就默认了应用已经存在。

## Minimum Inputs For One-Shot Execution

- 不需要微信登录；需要的是飞书开放平台登录态。
- 想实现“用户只给我最少信息，我一次性配好”，应区分两种模式：
  - 已有应用模式：
    - 提供 `App ID`
    - 提供 `App Secret`
    - 同一台机器上的浏览器已登录飞书开放平台，并且当前账号对该应用有编辑和发布权限
  - 从零创建模式：
    - 同一台机器上的浏览器已登录飞书开放平台
    - 当前账号有创建企业自建应用、编辑应用、创建版本和发布版本的权限
    - 允许默认生成应用名称、描述、图标占位或直接复用配置文件导入
- 如果是国际版 Lark，还需要能判断域名是 `lark` 而不是 `feishu`。

## One-Shot Support Boundary

- 在以下条件成立时，可以按“一次性执行”处理：
  - 浏览器里已有飞书开放平台控制台登录态
  - 当前账号有能力创建企业自建应用，或至少能编辑和发布已有应用
  - 当前账号有能力启用机器人、配置长连接、添加事件、创建版本、发布版本
  - 允许默认走官方支持路径：控制台创建应用，或基于配置文件导入创建应用
- 在以下条件下，不能把“只给 ID/Secret”视为足够：
  - 应用还不存在
  - 没有飞书开放平台登录态
  - 当前账号没有该应用的发布权限
  - 企业侧还有额外审批、安装或管理员策略阻塞

## Human-Only Boundary

- 以下环节不应承诺为“完全自动化”：
  - 用户首次登录飞书开放平台
  - 短信验证码、扫码登录、二次验证
  - 企业管理员审核或安全策略确认
  - 需要租户管理员手动放行的敏感权限和安装动作
- 也就是说：
  - “创建应用、配置能力、配置事件、创建版本、发布申请”可以尽量自动化
  - “替用户登录”和“替管理员完成审核”不能承诺自动化

## One-Click UX Contract

- 如果用户要“一键直达”，优先按以下体验定义执行：
  - 用户先把飞书开放平台页面打开并保持登录
  - 若已有应用，则给 `App ID` 和 `App Secret`
  - 若还没有应用，则直接走“从零创建模式”，自动创建一个企业自建应用
  - 其余步骤默认全部由代理继续完成，不再来回确认细枝末节
- 如果用户没有提供应用名称等非关键元数据：
  - 默认生成一个可工作的应用名称和描述
  - 优先保证链路可用，再允许用户后续回到后台补美化信息

## Minimum Target State

- 本地 OpenClaw 已配置并探活成功：
  - `openclaw gateway health`
  - `openclaw channels status --probe --json`
- 飞书线上版本至少包含：
  - `im.message.receive_v1`
  - 可选：`im.chat.access_event.bot_p2p_chat_entered_v1`
- 推荐保留的消息相关权限：
  - `im:message`
  - `im:message.p2p_msg:readonly`
  - `im:message.group_at_msg:readonly`
  - `event:ip_list`
  - `im:chat:readonly`
  - `im:chat.access_event.bot_p2p_chat:read`
- 只有需要接收群内所有消息时，才额外考虑敏感权限 `im:message.group_msg`

## Workflow

1. 先查文档
- 先确认 OpenClaw 官方要求的是哪种飞书接入模式。
- 再确认飞书官方对机器人、自建应用、长连接和发布的要求。

2. 先判断根因在哪一侧
- 如果 `openclaw gateway health` 和 `channels status --probe` 都是 `ok`，优先怀疑飞书应用侧没有发布线上版本。
- 如果飞书控制台里事件已经加上，但公开接口里线上版本没有事件，说明只是“变更已存在”，还没有真正发布。

3. 优先决定是“已有应用模式”还是“从零创建模式”
- 如果用户已经给了 `App ID` 和 `App Secret`，优先复用已有应用。
- 如果用户没有现成应用，但浏览器已登录开放平台，优先按官方支持路径创建企业自建应用。
- 如果能使用配置文件导入或官方教程提供的 manifest/import 能力，优先用它减少 UI 点击与漏配概率。

4. 再做本地配置
- 把飞书通道写入本地配置。
- 推荐先把私聊和群聊策略设清楚，再看消息链路。

5. 再做飞书应用配置与发布
- 如果应用尚未创建，先创建企业自建应用并拿到 `App ID` / `App Secret`。
- 启用机器人能力。
- 配置消息相关权限。
- 配置事件订阅方式为长连接，并添加 `im.message.receive_v1`。
- 优先走飞书开发者后台的正常发布路径。
- 如果 UI 被历史权限污染卡住，先判断是不是“保存按钮灰掉但草稿已具备可发布内容”。
- 不要把“页面灰掉”直接等同于“链路未配置”。

6. 最后做端到端验收
- 验证公开接口里线上版本已经切到新版本。
- 验证线上版本的事件列表里确实存在 `im.message.receive_v1`。
- 再去飞书里做真实私聊或群 `@` 测试。

## Practical Checks

- 本地健康检查：
  - `openclaw gateway health`
  - `openclaw channels status --probe --json`
- 飞书公开接口检查重点：
  - 当前 `online_version_id`
  - 线上版本号
  - 线上版本的 `event_infos`
- 如果线上版本仍是旧版，即使草稿里已经有长连接和事件，机器人仍可能没有反应。

## Supported Path First

- 首选飞书支持的路径：
  - 控制台创建企业自建应用
  - 如可用，优先通过配置文件导入创建应用
  - 控制台里启用机器人能力
  - 控制台里配置权限
  - 控制台里配置长连接
  - 添加事件
  - 创建版本
  - 发布版本
- 只有在 UI 明显被卡死、且用户要求“一次性修通”时，才进入 Atlas 页内 same-origin 兜底。

## Atlas Fallback

- Atlas 兜底是最后手段，不是默认手段。
- 只在以下情况使用：
  - 飞书控制台页面确实已加载
  - 用户明确允许在该页面内继续操作
  - 官方支持路径已经确认但 UI 无法完成最后一步
- 兜底细节见：
  - `references/checklist.md`
  - `references/atlas-fallback.md`

## Output Contract

执行该 Skill 时，输出至少应包含：

1. 当前问题在本地还是飞书应用侧的判断
2. 当前属于“已有应用模式”还是“从零创建模式”
3. 哪些步骤可自动化完成，哪些步骤仍属于人工边界
4. 是否“只差发布线上版本”的结论
5. 实际生效的线上版本号和事件列表
6. 本地 OpenClaw 探活结果
7. 用户接下来在飞书里的最终使用方式
