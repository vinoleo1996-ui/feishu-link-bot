# feishu-link-bot

把 OpenClaw 正式接入飞书自建应用机器人，并尽量做到“一键直达”。

这个 Skill 不是临时桥接方案，目标是走飞书官方支持路径，把下面这些事情一次性做完：

- 创建企业自建应用，或复用已有应用
- 启用机器人能力
- 配置消息相关权限
- 配置长连接事件订阅
- 添加 `im.message.receive_v1`
- 创建版本并发布到线上
- 配置本地 OpenClaw
- 做端到端验收

## 适用场景

- 你已经有飞书应用，只想把 OpenClaw 接进去
- 本地 OpenClaw 已经配好，但飞书机器人完全没反应
- 你想把“只要登录开放平台，剩下自动完成”固化成稳定工作流

## 使用方法

### 方式一：已有应用模式

你提供：

- `App ID`
- `App Secret`

同时满足：

- 当前机器浏览器已经登录飞书开放平台
- 当前账号对这个应用有编辑和发布权限

直接让 Codex 调用：

```md
$feishu-link-bot 帮我把 OpenClaw 接到这个飞书机器人里，直接配到可用并验收
```

### 方式二：从零创建模式

你只需要先把飞书开放平台页面打开并保持登录。

只要当前账号能创建企业自建应用、编辑应用、创建版本和发布版本，Codex 就可以继续：

- 自动创建企业自建应用
- 默认生成可工作的应用名称和描述
- 启用机器人与权限
- 配置长连接与事件
- 发布版本并做验收

直接让 Codex 调用：

```md
$feishu-link-bot 我还没有应用，直接从零创建并接到 OpenClaw，可用后告诉我怎么在飞书里发消息
```

## 一键体验边界

这个 Skill 支持“条件式一键”，不是“绝对无条件全自动”。

可以自动化的部分：

- 创建应用
- 配置能力
- 配置权限
- 配置事件
- 创建版本
- 提交发布
- 本地 OpenClaw 配置
- 验收与排障

不能承诺完全自动化的部分：

- 首次登录飞书开放平台
- 短信验证码、扫码登录、二次验证
- 企业管理员审核
- 需要管理员放行的敏感权限

## 注意事项

- 先查官方文档，再动手。
- 不要把“本地探活成功”误判成“飞书侧已经可用”。
- 飞书控制台里已经加了事件，不代表线上版本已经生效。
- 如果线上版本里没有 `im.message.receive_v1`，机器人大概率不会回。
- 优先走官方支持路径；只有控制台 UI 明显卡死时，才考虑 Atlas 页内 same-origin 兜底。
- GitHub、社区和第三方仓库只能补充实操坑位，不能覆盖官方结论。
